3. Was in reta Constraint Programming sein sollte
Nur der Teil, wo du keine einfache direkte Ableitung mehr hast, sondern ein Lösungsproblem.
Also dort, wo mehrere Bedingungen zugleich gelten und nicht klar ist, welche Lösung oder Kombination daraus folgt.

A. Parameterkompatibilität
Sehr geeigneter Bereich:
welche Kombination von Parametern ist zulässig
welche Kombination ist widersprüchlich
welche Parameter implizieren oder verbieten andere
welche Werte dürfen gleichzeitig auftreten
Das ist echtes Constraint-Thema.

B. Spaltenauswahl bei Mehrdeutigkeit
Wenn ein Request nicht genau auf eine Spaltenmenge zeigt, sondern:
mehrere Kandidaten zulässt
exakte und nicht exakte Treffer hat
Prioritäten berücksichtigt
Konflikte auflösen muss
dann ist Constraint Programming sehr passend.

C. Reverse Mapping
Das ist fast der klassische CP-Fall.
Beispiel:
gegebene Ausgabe oder Spaltenmenge
finde mögliche Requests, Generatoren oder Parameterkombinationen, die dazu führen
Das ist Suche unter Bedingungen, also Constraint-nah.

D. Konfliktauflösung und beste Auswahl
Wenn du sagst:
muss diese Eigenschaften erfüllen
sollte möglichst jene anderen Eigenschaften erfüllen
darf nicht mit diesen Regeln kollidieren
bevorzuge exakte Äquivalenz gegenüber bloßer Ähnlichkeit

dann bist du schon im Bereich von:
Constraints
eventuell Ranking
eventuell Optimierung
E. Konsistenzprüfung komplexer Modellteile

Zum Beispiel:
jede Unterkategorie muss zu genau einer Oberkategorie gehören
jede deklarierte Projektion muss für alle relevanten Typen definiert sein
jede Formatregel muss auf existierende semantische Objekte zeigen
keine zirkulären Verbote in einer bestimmten Klasse von Parametern
Das kann mit Constraint-Denken sehr sauber geprüft werden.

def project(table: RelTable, selected: tuple[str, ...]) -> RelTable:
    new_rows = tuple(
        RelRow(values={k: row.values[k] for k in selected})
        for row in table.rows
    )
    return RelTable(name=f"{table.name}_project", columns=selected, rows=new_rows)


def restrict(table: RelTable, predicate) -> RelTable:
    return RelTable(
        name=f"{table.name}_restrict",
        columns=table.columns,
        rows=tuple(row for row in table.rows if predicate(row))
    )


def simple_join(left: RelTable, right: RelTable, left_key: str, right_key: str) -> RelationView:
    joined = []
    for lrow in left.rows:
        for rrow in right.rows:
            if lrow.values[left_key] == rrow.values[right_key]:
                merged = dict(lrow.values)
                merged.update(rrow.values)
                joined.append(RelRow(values=merged))
    return RelationView(
        name=f"{left.name}_{right.name}_join",
        source_tables=(left.name, right.name),
        rows=tuple(joined),
    )

1. Garben (Sheaves) in reta
Intuition (ohne Theorieballast)

Eine Garbe sagt:

Lokale Informationen müssen sich konsistent zu globaler Information zusammensetzen lassen.

Teil-Requests
Teil-Spalten
Teil-Generatoren

Lokal:
einzelne Parameter:
--zeit=heute
--typ=sonne
--primzahlen=innenalle
Global:
vollständiger Request

Wenn mehrere Teil-Requests kompatibel sind:

R1: --zeit=heute
R2: --typ=sonne
R3: --primzahlen=innenalle

→ dann muss es eine eindeutige globale Zusammenfügung geben:

R = Kombination(R1, R2, R3)
Anwendung in reta
